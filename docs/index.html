How it works

Five typed phases. Each one fails closed before the next can start — no silent skips, no half-deployed agents.

  1. phase 1 📄

    Parse

    Strict Draft 2020-12 schema. Defaults filled. Config frozen.

  2. phase 2 🎯

    Generate

    Streams grok-4.20-0309 via the official xai-sdk. Writes manifest.json.

  3. phase 3 🛡️

    Safety

    Static regex sweep + JSON-mode Grok audit. Fails closed.

  4. phase 4 🚀

    Deploy

    Dispatch on deploy.target: x · vercel · render · local.

  5. phase 5 ✅

    Result

    BridgeResult: path · verdict · URL · cost · duration.

Full pipeline runs in ~40 seconds. Use --dry-run to exercise phases 1–3 without deploying.

🎭 Orchestra-native

Two reviewers.
One durable contract.

Grok Agent Orchestra runs the multi-agent debate and applies the Lucas veto — an epistemic gate that blocks unsourced, biased, or harmful briefs before Bridge ever sees them.

Bridge's two-layer code audit then handles the executable. A release ships only when both gates pass. Defence in depth: the probability of a false-pass surviving all three reviewers is the product of three small numbers.

Explore Orchestra → See the worked example
bridge.yaml 
version: "1.0"
name: weekly-research-publisher
description: Publishes the Orchestra-vetted brief as a weekly X thread.

build:
  source: grok
  grok_prompt: |
    Read out/research-brief.md and render a 7–10 post X thread...

deploy:
  target: x
  schedule: "0 13 * * 1"     # Mondays 13:00 UTC
  post_to_x: true

safety:
  audit_before_post:   true
  lucas_veto_enabled:  true     # ← one line wires Orchestra in
